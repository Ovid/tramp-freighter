## Executive Summary

The mobile responsive implementation is well-structured and closely follows the design plan. The CSS-first approach with JS hooks for structural changes is clean. However, three verified important issues exist: a severity classification divergence between mobile and desktop HUDs, a UX-impacting side effect where expanding the mobile HUD clears the user's star system selection, and fuel percentage display that breaks with ship upgrades. One critical pre-existing bug in the upgrade system was also surfaced by this review.

## Critical Issues

### [C1] Upgrade effects applied additively but values are absolute targets

- **File:** `src/game/constants.js` (upgrade effects) and `src/game/state/managers/ship.js:489`
- **Bug:** The upgrade application code uses `+=` to apply effects, but the effect values in constants are absolute targets. Extended Fuel Tank has `fuelCapacity: 150` (comment: "Up from 100"), yielding 100 + 150 = 250 instead of 150. Reinforced Hull has `cargoCapacity: 45` (comment: "Down from 50"), yielding 50 + 45 = 95 instead of 45.
- **Impact:** Fuel capacity is 250% instead of 150% after upgrade. Cargo capacity is 95 instead of 45 after hull upgrade. Game balance is significantly affected. This also compounds the fuel display bug (F1 below).
- **Suggested fix:** Either change effect values to deltas (`fuelCapacity: 50`, `cargoCapacity: -5`) or change the application logic from `+=` to `=`.
- **Confidence:** High
- **Found by:** Logic & Correctness
- **Note:** This is a pre-existing bug, not introduced by the mobile branch, but surfaced during review.

## Important Issues

### [I1] getSeverityClass diverges from getConditionClass — mobile shows fewer severity tiers

- **File:** `src/features/hud/MobileHUD.jsx:26-32`
- **Bug:** `getSeverityClass` classifies conditions into 3 tiers (ok/warning/critical), while the existing `getConditionClass` in `src/game/utils/string-utils.js:77-82` uses 4 tiers (good/fair/poor/critical). A ship system at 30% health shows as "ok" (green) on mobile but "poor" (red/orange) on desktop. The thresholds are: mobile considers < 50 as "warning" and < 25 as "critical"; desktop considers < 75 as "fair", < 50 as "poor", < 25 as "critical".
- **Impact:** Mobile users get less granular safety feedback. A system at 30% (genuinely degraded) appears healthy on mobile.
- **Suggested fix:** Reuse `getConditionClass` from string-utils and map its return values: `{ good: 'ok', fair: 'ok', poor: 'warning', critical: 'critical' }`.
- **Confidence:** High
- **Found by:** Contract & Integration, Logic & Correctness

### [I2] Expanding mobile HUD clears star system selection as side effect

- **File:** `src/features/hud/MobileHUD.jsx:57-59` and `src/App.jsx:435-438`
- **Bug:** When the user taps the mobile HUD bar to expand it, `onDismissPanel()` is called, which executes `handleClosePanel()` AND `setViewingSystemId(null)`. A user viewing a star system's info who wants to check fuel before jumping will lose their system selection when expanding the HUD.
- **Impact:** Users must re-select stars after checking HUD, frustrating the jump planning workflow.
- **Suggested fix:** Either split `onDismissPanel` into separate callbacks, or only clear `activePanel` (not `viewingSystemId`) when expanding the HUD.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I3] Fuel value displayed as raw number with % suffix — breaks with upgrades

- **File:** `src/features/hud/MobileHUD.jsx:13-14,74`
- **Bug:** `getWorstResources` uses the raw fuel value (which can exceed 100 with Extended Fuel Tank upgrade) and displays it with a `%` suffix. At full tank with the upgrade, this shows "Fuel 250%" (or "Fuel 150%" with corrected upgrade values). The severity thresholds (POOR=25, FAIR=50) also compare against the raw value, not a percentage of capacity. Desktop `ShipStatus.jsx` has the same underlying issue.
- **Impact:** Misleading fuel percentage display and incorrect severity classification for upgraded ships.
- **Suggested fix:** Normalize fuel as `(fuel / fuelCapacity) * 100` before display and threshold comparison.
- **Confidence:** High
- **Found by:** Logic & Correctness
