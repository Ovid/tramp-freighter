## Executive Summary

The mobile responsive design implementation is well-structured with clean separation between mobile and desktop code paths. One Critical issue was found: the `useClickOutside` hook's allowlist doesn't include mobile HUD elements, causing unintended undocking when users tap the HUD bar while docked. Four Important issues cover star re-selection, toggle label inconsistency, dock workflow friction, and a stale starmap selection ring. No security concerns.

## Critical Issues

### [C1] useClickOutside allowlist misses MobileHUD — tapping HUD bar while docked triggers undock
- **File:** `src/hooks/useClickOutside.js:16` and `src/features/station/StationMenu.jsx:28`
- **Bug:** `useClickOutside` ignores clicks inside `#game-hud` (line 16), but on mobile the HUD renders as `MobileHUD` which uses class `.mobile-hud-bar` — there is no element with `id="game-hud"`. `StationMenu` uses `useClickOutside(ref, onUndock)` (line 28), so any tap on the mobile HUD bar that isn't inside the station menu ref will be treated as an "outside" click, triggering `onUndock` and removing the player from the station.
- **Impact:** On mobile, tapping the collapsed HUD bar (which is always visible above the station menu) will unexpectedly undock the player. This is a major gameplay-breaking bug — the HUD bar is the most prominent tap target on screen.
- **Suggested fix:** Either (a) add `.mobile-hud-bar` and `.mobile-hud-expanded` to the `useClickOutside` allowlist: `if (e.target.closest('.mobile-hud-bar, .mobile-hud-expanded, .mobile-hud-backdrop')) return;`, or (b) wrap the MobileHUD in an element with `id="game-hud"` so the existing allowlist works.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

## Important Issues

### [I1] MobileCameraToolbar star re-selection impossible
- **File:** `src/features/navigation/MobileCameraToolbar.jsx:42-46`
- **Bug:** The `<select value="" onChange={...}>` is controlled with a hardcoded empty string value. After selecting a star, React resets the value to `""`. The browser's `onChange` will not fire if the user selects the same `<option>` again because the DOM value hasn't changed.
- **Impact:** Users cannot re-navigate to the same star they just selected without picking a different star first. This is a reasonable use case (re-centering on a star).
- **Suggested fix:** Reset the select value after handling the change, e.g., by using `e.target.value = ""` in the handler after processing, or add a key counter that increments after each selection to force remount.
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

### [I2] Mobile toggle label "Rotation" inconsistent with desktop "Star Rotation"
- **File:** `src/features/navigation/CameraControls.jsx:87`
- **Bug:** The toggle key `showRotation` generates the label "Rotation" via `key.replace(/^show/, '').replace(/([A-Z])/g, ' $1').trim()`. On desktop (line 137), this setting is labeled "Star Rotation". The mechanical label generation loses the "Star" qualifier.
- **Impact:** Mobile users see "Rotation" while desktop users see "Star Rotation" for the identical setting. Ambiguous — could mean camera rotation, star rotation, or auto-rotation.
- **Suggested fix:** Either rename the key to `showStarRotation`, or use an explicit label map instead of string manipulation.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I3] panelActive blocks HUD expansion when system panel is open, adding friction to dock workflow
- **File:** `src/App.jsx:434-438` and `src/features/hud/MobileHUD.jsx:59`
- **Bug:** `panelActive` includes `showSystemPanel` (when `viewingSystemId !== null`). On mobile, tapping any star opens the system panel full-screen and `panelActive` becomes true, locking the HUD in collapsed mode. To dock, the user must: (1) close system panel, (2) expand HUD, (3) tap Dock. On desktop this is a single click since HUD and system panel coexist side by side.
- **Impact:** The most common gameplay loop (view system info -> dock) requires extra steps on mobile compared to desktop.
- **Suggested fix:** Consider not including `showSystemPanel` in `panelActive` on mobile (the expanded HUD has z-index 40, above panels at 20), or add a "Dock" button directly to the system panel when viewing the current system on mobile.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I4] Stale starmap selection ring after mobile dock
- **File:** `src/App.jsx:260-263`
- **Bug:** When docking on mobile, `setViewingSystemId(null)` closes the system panel but does not call `starmapRef.current.deselectStar()`. The 3D selection ring remains visible in the starmap. Compare with `handleCloseSystemPanel` (line 316) and `handleJumpComplete` (line 331), which both call `deselectStar()`.
- **Impact:** On mobile, the station menu covers the starmap so this is invisible while docked. But after undocking (via `handleUndock`, which also doesn't deselect), the selection ring from the pre-dock star selection persists — a confusing UX where a star appears selected but no system panel is open.
- **Suggested fix:** Add `starmapRef.current?.deselectStar()` in `handleDock` when clearing `viewingSystemId` on mobile, or in `handleUndock`.
- **Confidence:** Medium
- **Found by:** Contract & Integration
