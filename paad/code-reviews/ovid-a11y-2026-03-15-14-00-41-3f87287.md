## Important Issues

### [I1] Escape key in useClickOutside undocks player when sub-panel is open
- **File:** `src/hooks/useClickOutside.js:24-28`
- **Bug:** The new Escape handler in `useClickOutside` fires `onUndock` from `StationMenu` even when a sub-panel (Trade, Refuel, Repair, etc.) is open. The mousedown handler correctly checks `e.target.closest('[data-panel]')` before closing, but the Escape handler only guards against `.modal-overlay` — sub-panels render inside `<div data-panel>` without a `.modal-overlay`, so the guard doesn't help. Pressing Escape while viewing the Trade panel undocks the player entirely.
- **Impact:** Destructive keyboard shortcut — user loses station context unexpectedly.
- **Suggested fix:** Add a `[data-panel]` guard to the Escape handler mirroring the mousedown handler: `if (document.querySelector('[data-panel]:not([id="station-interface"])')) return;`
- **Confidence:** High
- **Found by:** Contract & Integration (A)

### [I2] StarMapCanvas keyboard handler allocates Three.js objects per keypress
- **File:** `src/features/navigation/StarMapCanvas.jsx:429-433`
- **Bug:** `new THREE.Vector3()` and `new THREE.Spherical()` are created on every arrow key press inside `handleKeyDown`. CLAUDE.md states: "Never create objects in hot loops (animation frames, frequent events)." With key-repeat, this fires 20-30+ times/second. The same component's animation loop (line 256) already pre-allocates a `_tempOffset` to avoid this exact problem.
- **Impact:** GC pressure causing frame-time spikes during keyboard navigation, inconsistent with the project's own established pattern.
- **Suggested fix:** Hoist reusable `Vector3` and `Spherical` instances to module-level or ref scope.
- **Confidence:** High
- **Found by:** Logic & Correctness (A), Error Handling (A), Security

### [I3] SystemPanel critical-damage modal lacks focus trap and Escape handler
- **File:** `src/features/navigation/SystemPanel.jsx:132-151`
- **Bug:** The critical-damage modal is a hand-rolled `<div className="modal-overlay">` with `role="alertdialog"` and `aria-modal="true"`, but has no focus trap, no Escape handler, and no initial focus management. The project has a `Modal` component with all of these features that is not used here. Keyboard users cannot dismiss with Escape and focus can escape the dialog.
- **Impact:** `aria-modal="true"` promises behavior that isn't implemented, creating a false a11y contract. Keyboard users must Tab to the "Understood" button.
- **Suggested fix:** Use the existing `Modal` component, or add focus trap + Escape handler.
- **Confidence:** High
- **Found by:** Logic & Correctness (A)

### [I4] useNotification dismissNotification race with in-flight fadeOutAndRemove
- **File:** `src/hooks/useNotification.js:186-198`
- **Bug:** When the auto-dismiss timer fires, it calls `fadeOutAndRemove(id, processQueue)` which sets `fadeTimerRef.current` to a 300ms timeout. If the user clicks dismiss during that 300ms window, `dismissNotification` calls `fadeOutAndRemove` again — overwriting `fadeTimerRef.current` and leaking the first timeout. Both callbacks call `processQueue()`, potentially double-advancing the notification queue and showing duplicate notifications.
- **Impact:** Queue invariant (one notification at a time) can be violated. Duplicate notifications may appear.
- **Suggested fix:** Guard `fadeOutAndRemove` or `dismissNotification` to skip if the notification is already in `fadeOut: true` state.
- **Confidence:** Medium
- **Found by:** Error Handling (A), Concurrency & State (A)
