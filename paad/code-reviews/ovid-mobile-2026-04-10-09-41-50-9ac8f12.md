## Important Issues

### [I1] Upgrade delta values create save compatibility gap for cached ship capacities
- **File:** `src/game/constants.js:324,334,352,380` and `src/game/state/game-coordinator.js:606`
- **Bug:** Upgrade effect values were correctly fixed from buggy absolute values to proper deltas (e.g., `extended_tank.fuelCapacity`: 150->50, `expanded_hold.cargoCapacity`: 75->25). The old values were applied additively via `+=` in `calculateShipCapabilities()`, producing wrong totals (e.g., 100+150=250 instead of intended 150). The fix is correct. However, `restoreState()` loads `ship.fuelCapacity` and `ship.cargoCapacity` directly from the save without recalculation. On load of an old save with purchased upgrades, the UI (via EVENT_STATE_MAP event emission) shows the stale cached capacity while runtime methods like `getFuelCapacity()` return the recalculated correct value. This creates a desync between displayed and actual capacity.
- **Impact:** Players with existing saves who purchased `extended_tank`, `expanded_hold`, `reinforced_hull`, or `medical_bay` would see incorrect capacity values in the UI after loading until the next upgrade operation triggers recalculation. In the worst case, the UI might show 250 fuel capacity while the game logic caps refueling at 150.
- **Suggested fix:** Add a recalculation step in `_applyMigrations()` or `restoreState()` that calls `calculateShipCapabilities()` after load and updates the cached `ship.fuelCapacity` and `ship.cargoCapacity` values to match.
- **Confidence:** High
- **Found by:** Contract & Integration

## Suggestions

### [S1] MobileCameraToolbar settings popover has no click-outside dismiss
- **File:** `src/features/navigation/MobileCameraToolbar.jsx:14,56`
- **Bug:** The settings popover can only be dismissed by tapping the gear button again. Desktop `CameraControls` uses `useClickOutside` (line 49) for this purpose. The popover persists in state behind fullscreen panels and reappears when those panels close.
- **Suggested fix:** Add `useClickOutside` or a transparent backdrop (like MobileHUD uses) to dismiss the popover on outside tap.
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

### [S2] MobileHUD silently ignores tap when collapsed + panel active, no user feedback
- **File:** `src/features/hud/MobileHUD.jsx:58-61`
- **Bug:** When `panelActive` is true and the HUD is collapsed, tapping the bar does nothing silently. No `aria-disabled`, no visual disabled state, and the `aria-label` still says "Expand HUD".
- **Suggested fix:** Add `aria-disabled="true"` and a visual indicator (e.g., reduced opacity) when the HUD cannot expand, or allow expanding over panels since the HUD has higher z-index.
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [S3] handleDock clears viewingSystemId only on mobile -- asymmetry with handleUndock
- **File:** `src/App.jsx:258-260`
- **Bug:** `handleDock` clears `viewingSystemId` only when `isMobile` is true, but `handleUndock` (line 268) clears it unconditionally. This appears intentional (mobile has no room for side panels), but the asymmetry is undocumented.
- **Suggested fix:** Add a brief comment explaining the mobile-only clearing rationale.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [S4] CSS mobile fullscreen pattern repeated in multiple blocks
- **File:** `css/variables.css:238-305`, `css/hud.css:716-730`
- **Bug:** Three distinct `@media (max-width: 600px)` blocks apply nearly identical fullscreen positioning CSS (position:fixed, top:var(--mobile-hud-height), width:100%, etc.). Changes to `--mobile-hud-height` semantics would require updating all blocks in sync.
- **Suggested fix:** Consider a shared `.mobile-fullscreen` utility class for the common properties, with per-panel overrides only for differing properties (e.g., safe-area padding, z-index).
- **Confidence:** Medium
- **Found by:** Contract & Integration
