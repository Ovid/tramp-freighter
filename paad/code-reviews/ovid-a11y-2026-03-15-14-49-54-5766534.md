## Executive Summary

This a11y branch is a thorough accessibility pass with well-executed ARIA attributes, focus management, keyboard navigation, and reduced motion support. One critical bug was found: hull breach and life support mechanical failures can be dismissed with zero game consequences due to the Acknowledge button bypassing encounter resolution. Six important issues were identified, mostly around keyboard navigation edge cases and encounter panel close buttons silently triggering game-consequential actions while labeled as "Close" for screen readers.

## Critical Issues

### [C1] MechanicalFailurePanel hull_breach/life_support Acknowledge bypasses game outcome
- **File:** `src/features/danger/MechanicalFailurePanel.jsx:408`
- **Bug:** For `hull_breach` and `life_support` failure types, the "Acknowledge" button calls `onClose` which maps to `handleEncounterClose` in the orchestration hook. This clears the encounter UI without calling `resolveEncounter` or `applyEncounterOutcome`. The hull damage (`costs.hull`, `costs.cargoLoss`) and life support degradation (`costs.lifeSupport`) are never applied to game state.
- **Impact:** Players can dismiss hull_breach and life_support events with zero consequences. These are supposed to be severe penalties.
- **Suggested fix:** The Acknowledge button should call `onChoice('acknowledge')` instead of `onClose`. The orchestration hook should route `'acknowledge'` for `mechanical_failure` through `resolveEncounter`, then display the outcome via `OutcomePanel` before closing.
- **Confidence:** High
- **Found by:** Contract & Integration B

## Important Issues

### [I1] Focus management targets non-existent CSS selectors for endgame modes
- **File:** `src/App.jsx:163`
- **Bug:** `document.querySelector('.pavonis-run, .epilogue')` queries by class, but both `PavonisRun` and `Epilogue` components use `id="pavonis-run"` and `id="epilogue"` on their root divs. The query returns null, so focus is never transferred when entering PAVONIS_RUN or EPILOGUE modes.
- **Impact:** Keyboard users entering endgame sequences lose focus context. Screen readers are not informed of the view change.
- **Suggested fix:** Change the selector to `'#pavonis-run, #epilogue'`.
- **Confidence:** High
- **Found by:** Contract & Integration A

### [I2] MissionCompleteNotifier currentIndex race condition
- **File:** `src/features/missions/MissionCompleteNotifier.jsx:23-38`
- **Bug:** `completeMission` fires `MISSIONS_CHANGED`, which triggers the `useEffect` listening on `missions`. That effect unconditionally calls `setCurrentIndex(0)`, which races against the `setCurrentIndex(nextIndex)` already scheduled by `handleComplete`. The index resets to 0, causing the notifier to always show the first completable mission repeatedly rather than stepping through the queue.
- **Impact:** With multiple completable missions, clicking "Claim Reward" may loop on the same mission instead of advancing.
- **Suggested fix:** Remove `setCurrentIndex(0)` from the `useEffect`. Only reset to 0 when the completable mission ID set actually changes, not on every missions event.
- **Confidence:** High
- **Found by:** Concurrency & State B

### [I3] CombatPanel/PirateEncounterPanel evasive probability divergence
- **File:** `src/features/danger/CombatPanel.jsx:521-528` vs `src/features/danger/PirateEncounterPanel.jsx:404-421`
- **Bug:** `calculateCombatProbabilities` applies a `-0.1` engine condition penalty to evasive chance; `calculateTacticalProbabilities` omits this check entirely. Both display "evasive" probability to the player for the same encounter at different phases.
- **Impact:** At the initial pirate encounter, a player with a degraded engine sees inflated evasive odds. When combat escalates, the probability shown drops by 10%. Misleading information that can influence player decisions.
- **Suggested fix:** Apply the same engine condition penalty in `calculateTacticalProbabilities`, or extract the shared calculation to a single utility function.
- **Confidence:** High
- **Found by:** Contract & Integration B

### [I4] Encounter panel close buttons trigger game-consequential choices labeled "Close"
- **Files:** `src/features/danger/PirateEncounterPanel.jsx:107`, `CombatPanel.jsx:105`, `NegotiationPanel.jsx:80`, `InspectionPanel.jsx:66`, `DistressCallPanel.jsx:76`
- **Bug:** The close (x) button on all five encounter panels calls `onChoice('flee')` or `onChoice('ignore')` while carrying `aria-label="Close"`. These fire game-consequential choices (evasive maneuvers, ignore distress call) without the user knowing.
- **Impact:** Keyboard and screen reader users pressing the close button unknowingly trigger flee/ignore outcomes. `flee` can result in hull damage or combat; `ignore` applies karma penalties.
- **Suggested fix:** Either label these buttons with their actual action (`aria-label="Flee"`, `aria-label="Ignore distress call"`) or replace them with a visible labeled button in the action area.
- **Confidence:** High
- **Found by:** Contract & Integration B, Logic & Correctness B

### [I5] useNotification dismiss during auto-fade stalls queue permanently
- **File:** `src/hooks/useNotification.js:26-37, 188-199`
- **Bug:** `fadeOutAndRemove` has a guard: `if (fadeTimerRef.current !== null) return`. If the auto-dismiss timer fires and starts the fade, then the user clicks the dismiss button before the fade completes, `dismissNotification` calls `fadeOutAndRemove` which hits the guard and returns. `processQueue()` is never called by the dismiss path. The queue stalls: `isShowingRef.current` remains `true` forever.
- **Impact:** If a user dismisses a notification mid-fade, the notification queue is permanently stalled. No further notifications appear for the rest of the session.
- **Suggested fix:** In `dismissNotification`, if `fadeTimerRef.current` is already set, cancel the existing fade timer, proceed with immediate removal, and call `processQueue`.
- **Confidence:** High
- **Found by:** Concurrency & State A, Error Handling A

### [I6] useClickOutside Escape key always bails when at station
- **File:** `src/hooks/useClickOutside.js:28-33`
- **Bug:** The Escape handler queries `[data-panel]:not(#camera-controls)` and returns early if any found panel is not `ref.current`. At the station, `StationMenu` and `PanelContainer` both have `data-panel` and are always in the DOM. Any panel's `useClickOutside` will find these other `data-panel` elements, hit the guard, and return without firing `onCloseRef.current()`.
- **Impact:** Keyboard users cannot close panels via Escape while docked at a station. This breaks a core keyboard navigation pattern that the a11y branch explicitly adds.
- **Suggested fix:** The guard should check if the other `data-panel` element is a *higher-priority* overlay, not just any other panel in the DOM. Simplest fix: only suppress if a panel other than `ref.current` is open AND is a descendant of a different hierarchy (e.g., check if the active focus is inside `ref.current`).
- **Confidence:** High
- **Found by:** Logic & Correctness A, Contract & Integration A, Error Handling A
