## Important Issues

### [I1] Escape key double-fire between Modal and useClickOutside
- **File:** `src/hooks/useClickOutside.js:24-27` and `src/components/Modal.jsx:62-66`
- **Bug:** Both register document-level `keydown` listeners for Escape. When a Modal is open inside a component using `useClickOutside` (confirmed: StationMenu uses `useClickOutside(ref, onUndock)` and renders ConfirmModals), a single Escape press fires both handlers. The useClickOutside `mousedown` handler has a `.modal-overlay` guard (line 19), but the `keydown` handler does not. Neither handler calls `event.stopPropagation()`.
- **Impact:** Pressing Escape while a modal is open in the station could close the modal AND undock simultaneously. The user loses station context unexpectedly.
- **Suggested fix:** Add a modal-overlay guard to the `useClickOutside` keydown handler: `if (document.querySelector('.modal-overlay')) return;` — mirroring the existing mousedown guard.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State

### [I2] OutcomePanel `getKarmaClass` uses wrong thresholds
- **File:** `src/features/danger/OutcomePanel.jsx:393-399`
- **Bug:** Uses hard-coded thresholds `25/-25` for GOOD/BAD karma boundaries, while `KARMA_CONFIG.DISPLAY_THRESHOLDS` (in `src/game/constants.js`) defines `GOOD: 20, BAD: -20`. NegotiationPanel and DistressCallPanel correctly use the constants. A player with karma 22 sees "good" in NegotiationPanel but "neutral" in OutcomePanel.
- **Impact:** Inconsistent karma classification across encounter panels. Players see contradictory karma status depending on which panel they're viewing.
- **Suggested fix:** Replace hard-coded thresholds with `KARMA_CONFIG.DISPLAY_THRESHOLDS` constants, matching NegotiationPanel's implementation.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I3] OutcomePanel `getReputationClass` uses wrong thresholds and different CSS classes
- **File:** `src/features/danger/OutcomePanel.jsx:407-413`
- **Bug:** Uses hard-coded thresholds `50/25/-25/-50` and returns CSS classes `'very-good', 'good', 'neutral', 'bad', 'very-bad'`, while the other three panels use `REPUTATION_BOUNDS` constants (thresholds `60/30/10/-10/-50`) and return `'trusted', 'friendly', 'warm', 'neutral', 'cold', 'hostile'`. A player at reputation 55 sees "trusted" in NegotiationPanel but "good" in OutcomePanel.
- **Impact:** Inconsistent reputation display across encounter outcome vs. encounter panels. Different CSS class names also mean different styling.
- **Suggested fix:** Use `REPUTATION_BOUNDS` constants and align CSS class names with the other panels.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I4] NegotiationPanel and InspectionPanel `getReputationTier` uses hard-coded magic numbers
- **File:** `src/features/danger/NegotiationPanel.jsx:511-519` and `src/features/danger/InspectionPanel.jsx:434-442`
- **Bug:** Both files hard-code reputation thresholds `90, 60, 30, 10, -10, -50` instead of using `REPUTATION_BOUNDS` constants. DistressCallPanel correctly uses the constants. While the values currently match, this violates the project's constants rule ("ALL magic numbers must go in constants.js") and will drift if REPUTATION_BOUNDS changes.
- **Impact:** Future constant changes won't propagate to these files, creating silent divergence.
- **Suggested fix:** Import and use `REPUTATION_BOUNDS` constants for threshold values. (Note: InspectionPanel's intentionally different tier names like 'Exemplary' vs 'Family' are fine.)
- **Confidence:** High
- **Found by:** Contract & Integration

If `clearNotifications()` is called during fade-out, the callback still fires and calls `processQueue()`. Practical impact is minimal (no-op queue check), but storing the timeout ID would make cleanup deterministic. *(Error Handling, Concurrency & State)*

- **[S3]** `src/components/Modal.jsx:50-51`: Focus trap selector `'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'` doesn't exclude `:disabled` elements. If all focusable elements are disabled, `focus()` silently fails and focus trap breaks. Add `:not([disabled])` to button/input selectors. *(Logic & Correctness, Error Handling)*

- **[S4]** `src/features/endgame/EndCredits.jsx:26-37`: The `window` keydown listener for Space remains active after credits finish. Pressing Space calls `handleTogglePause` which could call `anim.play()` on a cancelled animation, restarting the credits visually. Add guard: `if (!anim || anim.playState === 'finished' || scrollFinished) return;`. *(Logic & Correctness, Error Handling)*

- **[S5]** Encounter panels `PirateEncounterPanel.jsx:96`, `CombatPanel.jsx:93`, `NegotiationPanel.jsx:71`, `InspectionPanel.jsx:54`, `MechanicalFailurePanel.jsx:52`, `DistressCallPanel.jsx:67`, `OutcomePanel.jsx:53`: These 7 panels are modal overlays but lack `role="dialog"` and `aria-modal="true"`, unlike DangerWarningDialog which received dialog semantics in this branch. *(Contract & Integration)*

- **[S6]** `src/components/Modal.jsx:41-43`: `previousFocusRef.current.focus()` is called without checking if the element is still in the DOM. If a view mode change unmounts the previously-focused element, focus silently goes nowhere. Guard with `document.body.contains(previousFocusRef.current)`. *(Error Handling)*

- **[S7]** `src/features/navigation/StarMapCanvas.jsx:406`: `setAutoRotationEnabled(!autoRotationEnabled)` reads from closure instead of using functional updater. Double-click in same render batch won't toggle twice. Use `setAutoRotationEnabled(prev => !prev)`. *(Concurrency & State)*

- **[S8]** `src/features/info-broker/InfoBrokerPanel.jsx:281-287,300-306`: Tab `onKeyDown` handler is duplicated identically on both tab buttons. Extract to a shared function reference. *(Contract & Integration)*

- **[S9]** `src/App.jsx:153-159`: Focus management for ENCOUNTER view mode queries `.panel-base.visible` via `requestAnimationFrame`. Null guard exists, but if the panel hasn't rendered yet, focus silently fails. Consider moving initial focus to each encounter component's own mount effect. *(Error Handling)*
