## Executive Summary

This branch adds comprehensive mobile responsive support (detection hook, context, HUD, camera toolbar, full-screen panels, CSS variables). The architecture is clean and well-tested. One critical z-index stacking bug was found where the station menu renders behind the collapsed HUD bar on mobile. Three important functional gaps were identified: inaccessible Instructions/Achievements on mobile, a select widget that can't re-select the same star, and inconsistent currency formatting between mobile and desktop HUD displays.

## Critical Issues

### [C1] Station menu z-index renders behind collapsed HUD on mobile
- **File:** `css/hud.css:728`
- **Bug:** Mobile station interface uses `z-index: var(--z-camera-toolbar)` (value 10), while the collapsed HUD bar uses `--z-hud-collapsed` (value 30). The station menu renders behind the HUD bar. The recent commit `5f036fb` ("station sub-panels hidden behind station menu") suggests this area has active stacking issues.
- **Impact:** Station content at the top of the screen is obscured by the HUD bar. Players may not be able to interact with station header or close button.
- **Suggested fix:** Change `z-index: var(--z-camera-toolbar)` to `z-index: var(--z-panel-fullscreen)` for `#station-interface` on mobile.
- **Confidence:** High
- **Found by:** Plan Alignment, verified by Verifier

## Important Issues

### [I1] Instructions and Achievements unreachable on mobile
- **File:** `src/features/navigation/CameraControls.jsx:69-106`
- **Bug:** CameraControls renders `InstructionsModal` and `AchievementsModal` in the mobile branch (lines 96-104), but `MobileCameraToolbar` has no buttons to trigger them. The `showInstructions` and `showAchievements` state variables are permanently `false` on mobile. These modals are rendered but can never be opened.
- **Impact:** Mobile players cannot access Instructions or Achievements from the starmap view. Features are silently missing.
- **Suggested fix:** Pass `onShowInstructions` and `onShowAchievements` callbacks to `MobileCameraToolbar` and add buttons in the settings popover or toolbar.
- **Confidence:** High
- **Found by:** Logic, Error Handling, Contract & Integration

### [I2] Mobile star finder `<select>` cannot re-select the same star
- **File:** `src/features/navigation/MobileCameraToolbar.jsx:34-48`
- **Bug:** The `<select>` uses `defaultValue=""` (uncontrolled). After selecting a star, the browser retains that value. Re-selecting the same star fires no `onChange`. Desktop uses a controlled `value=""` (`CustomSelect` at `CameraControls.jsx:188`), which resets after each selection.
- **Impact:** On mobile, users cannot re-navigate to a star they already selected without first selecting a different star.
- **Suggested fix:** Make the select controlled: `value=""` instead of `defaultValue=""`.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I3] Credits formatting mismatch between mobile and desktop HUD
- **File:** `src/features/hud/MobileHUD.jsx:65` vs `src/features/hud/ResourceBar.jsx:21`
- **Bug:** MobileHUD collapsed bar displays `{credits ?? 0} CR` (plain number, "CR" suffix). ResourceBar (rendered in expanded HUD and desktop) displays `₡{credits.toLocaleString()}` (locale-formatted, "₡" prefix). On mobile, the collapsed bar shows "12500 CR" while the expanded overlay shows "₡12,500".
- **Impact:** Inconsistent currency display between collapsed and expanded mobile HUD, and between mobile and desktop.
- **Suggested fix:** Use `₡{(credits ?? 0).toLocaleString()}` in MobileHUD.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I4] Z-index scale values swapped vs design document
- **File:** `css/variables.css:90-93`
- **Bug:** Design doc specifies `--z-hud-collapsed: 20`, `--z-panel-fullscreen: 30`. Implementation has `--z-panel-fullscreen: 20`, `--z-hud-collapsed: 30` (swapped). The current implementation may be intentionally better (HUD always accessible above panels), but contradicts the documented design.
- **Impact:** If the plan is treated as source of truth, this is a deviation. The current stacking (HUD above panels) is arguably correct for usability, but the plan should be updated to match.
- **Suggested fix:** Either update the plan to match the implementation, or swap the values if the plan is correct.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I5] Endgame components render outside MobileProvider
- **File:** `src/App.jsx:583-594`
- **Bug:** `PavonisRun` (line 584) and `Epilogue` (line 592) render as siblings to the `MobileProvider` block (lines 413-581), not inside it. If any component in those trees ever calls `useMobile()`, it will crash with a missing context error.
- **Impact:** Latent crash risk. Currently safe (endgame components don't use `useMobile()`), but as mobile work continues, this is a likely trap.
- **Suggested fix:** Move `MobileProvider` to wrap the entire non-title content, or duplicate it around endgame components.
- **Confidence:** Medium
- **Found by:** Concurrency & State, Contract & Integration
