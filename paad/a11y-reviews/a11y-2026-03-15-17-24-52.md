# Accessibility Audit: Tramp Freighter Blues

**Date:** 2026-03-15 17:24:52
**Commit:** 3d878b6ac45f20945eebf0ccf82781c82d00de87
**Platform(s):** Web
**Tech stack:** React 18, Three.js, Vite, CSS custom properties, no component library (custom Button, Card, Modal, CustomSelect)
**Files audited:** 55 user-facing JSX components + 5 Three.js engine files + CSS
**Existing a11y tooling:** None (`eslint-plugin-jsx-a11y` not installed, no axe-core, no pa11y)
**Conformance target:** WCAG 2.2 AA via WCAG2ICT (AAA noted as recommendations)
**Platform guidelines referenced:** WAI-ARIA Authoring Practices, Game Accessibility Guidelines

## Executive Summary

The codebase has a solid structural foundation -- `lang="en"` on `<html>`, skip navigation link, landmark roles, focus management on view mode transitions, a `prefers-reduced-motion` CSS rule, and a well-implemented `Modal` component with focus trapping. However, that Modal component is not consistently used: 11 dialog-style panels declare `aria-modal="true"` without implementing focus trapping, which is the most pervasive serious issue. Additionally, interactive buttons (buy/sell/repair) lack accessible context, status messages are not announced in most panels, and color-only meaning in the 3D starmap wormhole lines creates a gameplay barrier for colorblind users. Estimated conformance: **partial A** (several Level A failures remain).

## Impact Summary by User Group

- **Screen reader users:** Can access most content but encounter significant friction: dialog panels lack focus traps, status messages are not announced, button context is missing for trade/repair operations, and label-value pairs throughout are not semantically associated. The 3D starmap is mitigated by the "Find Star" dropdown.
- **Low-vision users:** Fixed panel widths break at 400% zoom. Some contrast issues exist on ancillary text. No `prefers-contrast` support.
- **Colorblind users:** Wormhole fuel feasibility in the 3D starmap relies solely on color (red/yellow/green), creating a significant gameplay barrier. Several encounter panels use color-only severity indicators, partially mitigated by text labels.
- **Motor-impaired users (keyboard/switch):** Keyboard navigation works for most flows via the "Find Star" dropdown and station menus. Focus management gaps when opening/closing station panels cause keyboard users to lose position. Dialog panels without focus traps allow escape to background.
- **Cognitive and learning disabilities:** No re-accessible help from the HUD. Auto-dismissing notifications (2-3s) are not user-adjustable. No difficulty options. Click-outside-to-undock is an unexpected context change. Encounter close buttons trigger game actions (flee) that look like standard close buttons.
- **Deaf and hard-of-hearing users:** No current impact (game has no audio system).
- **Vestibular and photosensitive users:** CSS `prefers-reduced-motion` rule covers CSS animations, but Three.js jump animations and Web Animations API credits scroll do not check the preference. Antimatter mode toggles instant full-screen inversion.

## Serious Issues (Major Difficulty)

### [S1] Focus Traps Missing on 9 Encounter/Dialog Panels
- **File:** `src/features/danger/CombatPanel.jsx:97`, `PirateEncounterPanel.jsx:99`, `NegotiationPanel.jsx:72`, `InspectionPanel.jsx:59`, `MechanicalFailurePanel.jsx:53`, `DistressCallPanel.jsx:68`, `OutcomePanel.jsx:58`, `DangerWarningDialog.jsx:69`, `NarrativeEventPanel.jsx:74`
- **Platform:** Web
- **Barrier:** All 9 panels declare `role="dialog" aria-modal="true"` but implement no focus trapping. Tab key escapes into background starmap and HUD. The `Modal` component at `src/components/Modal.jsx:76-92` correctly implements Tab/Shift-Tab cycling, Escape handling, and focus save/restore -- but none of these panels use it.
- **Criterion:** WCAG 2.4.3 Focus Order -- Level A
- **Affects:** Keyboard and screen reader users. `aria-modal="true"` tells assistive tech background is inert, but keyboard focus reaches it, creating mismatch and confusion.
- **Fix:** Extract focus-trap logic from `Modal.jsx` into a reusable `useFocusTrap(containerRef)` hook and apply it to each panel's container ref. Or refactor panels to render inside `Modal`.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard & Motor, React Patterns (3/6 specialists)

### [S2] Focus Trap Missing on ShipNamingDialog
- **File:** `src/features/title-screen/ShipNamingDialog.jsx:39-45`
- **Platform:** Web
- **Barrier:** Declares `role="dialog" aria-modal="true"` with its own `modal-overlay` div but no focus trap (does not use the Modal component). `autoFocus` sets initial focus on the input, but Tab escapes the dialog.
- **Criterion:** WCAG 2.4.3 Focus Order -- Level A
- **Affects:** Keyboard users at game start. Tab escapes the naming dialog.
- **Fix:** Use the `Modal` component or apply `useFocusTrap`.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard & Motor, React Patterns (3/6 specialists)

### [S3] Selection State Not Exposed on Encounter Option Buttons
- **File:** `src/features/danger/CombatPanel.jsx:250`, `PirateEncounterPanel.jsx:194`, `NegotiationPanel.jsx:144`, `DistressCallPanel.jsx:216`, `MechanicalFailurePanel.jsx:152`
- **Platform:** Web
- **Barrier:** These panels use a "select-then-confirm" pattern where clicking an option sets `selectedOption` state and applies a CSS class `selected`, but no `aria-pressed`, `aria-selected`, or `aria-current` is set. Screen readers announce plain buttons with no indication of which is currently selected.
- **Criterion:** WCAG 4.1.2 Name, Role, Value -- Level A
- **Affects:** Screen reader users cannot tell which option is selected before confirming.
- **Fix:** Add `aria-pressed={selectedOption === 'optionName'}` to each option button.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard & Motor (2/6 specialists)

### [S4] Buy/Sell Buttons Lack Accessible Context
- **File:** `src/features/trade/TradePanel.jsx:243-273` (buy), `TradePanel.jsx:359-376` (sell)
- **Platform:** Web
- **Barrier:** Buy buttons read "Buy 1", "Buy 5", "Buy Max" and sell buttons read "Sell 1", "Sell All (3)" with no `aria-label` providing commodity context. Multiple goods produce repeated identical button labels.
- **Criterion:** WCAG 4.1.2 Name, Role, Value -- Level A
- **Affects:** Screen reader users cannot identify which commodity a buy/sell button operates on.
- **Fix:** Add `aria-label={`Buy 1 ${goodName}`}` to each button, or wrap each commodity section in a group with `aria-label` naming the commodity.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard & Motor (2/6 specialists)

### [S5] Repair Buttons Lack Accessible Context
- **File:** `src/features/repair/RepairPanel.jsx:259-314`
- **Platform:** Web
- **Barrier:** Repair buttons read "+10% (C45)", "+25% (C112)", "Full (C280)" for each system (Hull, Engine, Life Support) with no `aria-label` distinguishing which system they repair. Identical labels repeat across systems.
- **Criterion:** WCAG 4.1.2 Name, Role, Value -- Level A
- **Affects:** Screen reader users cannot distinguish Hull repair from Engine repair buttons.
- **Fix:** Add `aria-label={`Repair ${systemName} ${buttonText}`}`.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard & Motor (2/6 specialists)

### [S6] Station Panel Focus Not Managed on Open/Close
- **File:** `src/features/station/PanelContainer.jsx:25-57`, `src/features/station/StationMenu.jsx`
- **Platform:** Web
- **Barrier:** PanelContainer has zero focus management -- no `useEffect`, no `ref`, no focus calls. When a panel opens from StationMenu, focus stays on the station button. When a panel closes, focus does not return to StationMenu. App.jsx manages focus for view mode transitions but NOT for panel transitions within the STATION view.
- **Criterion:** WCAG 2.4.3 Focus Order -- Level A
- **Affects:** Keyboard users lose their place when opening/closing station panels.
- **Fix:** Focus the panel heading on open; return focus to the triggering StationMenu button on close.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard & Motor, React Patterns (3/6 specialists)

### [M2] Wormhole Lines Use Color-Only to Indicate Fuel Feasibility
- **File:** `src/game/engine/wormholes.js:17-28`, `src/game/constants.js:865-870`
- **Platform:** Web (Three.js)
- **Barrier:** Wormhole connections change color: green (sufficient fuel), yellow (warning), red (insufficient). No secondary visual indicator (dash pattern, thickness, label) accompanies the color. This is a primary gameplay navigation cue.
- **Criterion:** WCAG 1.4.1 Use of Color -- Level A
- **Affects:** Color-blind users (especially red-green, ~8% of males) cannot distinguish feasible from infeasible jump routes.
- **Fix:** Add secondary encoding: vary line thickness or use dashed lines for insufficient fuel. Three.js supports `LineDashedMaterial`.
- **Confidence:** High
- **Found by:** Visual & Color (1/6 specialists)

### [M3] Label-Value Pairs Not Programmatically Associated (20+ Components)
- **File:** `src/features/hud/ResourceBar.jsx:19-26`, `src/features/endgame/Epilogue.jsx:55-83`, `src/features/cargo/CargoManifestPanel.jsx:74-97`, `src/features/achievements/StatsSection.jsx:88-153`, and 16+ additional components
- **Platform:** Web
- **Barrier:** Throughout the app, label-value pairs use `<div><span class="label">X:</span><span class="value">Y</span></div>` with no semantic association (`<dl>/<dt>/<dd>`, tables, or ARIA). Screen readers read these as disconnected text fragments.
- **Criterion:** WCAG 1.3.1 Info and Relationships -- Level A
- **Affects:** Screen reader users hear labels and values as unrelated text in dense info panels.
- **Fix:** Use `<dl>/<dt>/<dd>` for stat lists. Consider a shared `<InfoRow>` component using definition list markup internally.
- **Confidence:** High
- **Found by:** Screen Reader (1/6 specialists)
