## Critical Issues

### [C1] `listAvailableIntelligence` reads wrong property names from event objects
- **File:** `src/game/game-information-broker.js:301-304`
- **Bug:** Code accesses `systemEvent.name`, `systemEvent.commodity`, `systemEvent.modifier` (singular), but real events from `createEvent()` have shape `{ id, type, systemId, startDay, endDay, modifiers }`. All three reads return `undefined`. The test at `tests/unit/information-broker-coverage.test.js:410` uses hand-crafted fake event objects matching the *wrong* shape, so the test passes while validating nothing real.
- **Impact:** When Advanced Sensors are installed, `listAvailableIntelligence` returns `{ name: undefined, commodity: undefined, modifier: undefined }` for systems with active events. The feature was designed to surface event information to the information broker UI but produces empty data. Any future UI consumer will display `undefined` values.
- **Suggested fix:** Access correct properties:
  ```js
  const eventTypeDef = EconomicEventsSystem.EVENT_TYPES[systemEvent.type];
  if (eventTypeDef) {
    result.event = {
      name: eventTypeDef.name,
      commodities: Object.keys(systemEvent.modifiers),
      modifiers: systemEvent.modifiers,
    };
  }
  ```
  Update the test data in `information-broker-coverage.test.js` to use real event shapes.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration

## Important Issues

### [I1] Save-compatibility bug: old `medical_emergency` events with sub-1.0 modifiers trigger premium pricing
- **File:** `src/game/game-trading.js:46-54` and `src/game/game-events.js:203-210`
- **Bug:** Before this branch, `medical_emergency` had modifiers `{ medicine: 2.0, grain: 0.9, ore: 0.9 }`. The `removeExpiredEvents` save-compatibility filter strips events with unknown *types* (like deleted `supply_glut`) but does NOT sanitize stale modifiers on known types. If a player loads an old save with an active `medical_emergency`, the event object still carries `grain: 0.9`. `getEventModifier()` returns `0.9`, which is `!== 1.0`, so `calculatePrice` returns the guaranteed-premium `getEventPrice('grain')` — turning what was a 10% price *reduction* into a massive price *increase*.
- **Impact:** Temporary pricing inversion for players loading pre-update saves with active `medical_emergency` events. Effect expires when the old event ends, but could cause confusion or unfair trades in the interim.
- **Suggested fix:** In `removeExpiredEvents`, after the type-existence check, sanitize modifiers to match the current `EVENT_TYPES` definition:
  ```js
  const validModifiers = EconomicEventsSystem.EVENT_TYPES[event.type].modifiers;
  for (const key of Object.keys(event.modifiers || {})) {
    if (!(key in validModifiers)) {
      delete event.modifiers[key];
    }
  }
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I2] Off-by-one: event display hidden on final active day
- **File:** `src/features/navigation/SystemPanel.jsx:338`
- **Bug:** The display condition `daysRemaining > 0` hides the event indicator when `endDay === currentDay` (`daysRemaining = 0`). But `removeExpiredEvents` uses `endDay < currentDay` (strict less-than), so the event is still active on that day and still affects pricing. The player pays event prices without seeing an event indicator.
- **Impact:** On the last day of an event, players see inflated prices at a destination but no explanation of why, losing a trading signal.
- **Suggested fix:** Change to `daysRemaining >= 0` and handle the zero case:
  ```jsx
  {destinationEventType && daysRemaining >= 0 && (
    ...
    <div className="event-time-remaining">
      {daysRemaining === 0
        ? 'Final day'
        : `${daysRemaining} day${daysRemaining !== 1 ? 's' : ''} remaining`}
    </div>
  )}
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling

### [I3] Bridge Pattern violation: direct `getState()` call in component
- **File:** `src/features/navigation/SystemPanel.jsx:230`
- **Bug:** `const currentDay = game.getState().player?.daysElapsed ?? 0;` calls `getState()` directly. CLAUDE.md states: "Components must NEVER call `GameCoordinator.getState()` directly." Every other component that needs `currentDay` uses `useGameEvent(EVENT_NAMES.TIME_CHANGED)`.
- **Impact:** The value won't update reactively when time changes. More importantly, it violates the project's core architectural invariant and sets a precedent. The hook call must be placed at the top level of the component (before any conditional returns) to satisfy React's rules of hooks.
- **Suggested fix:** Replace with `const currentDay = useGameEvent(EVENT_NAMES.TIME_CHANGED) ?? 0;` at the component's hook section (lines 31-39).
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State

### [I4] No subscription to `ACTIVE_EVENTS_CHANGED` — stale event data in SystemPanel
- **File:** `src/features/navigation/SystemPanel.jsx:312-317`
- **Bug:** `game.getActiveEventForSystem(system.id)` is called during render, but the component has no `useGameEvent(EVENT_NAMES.ACTIVE_EVENTS_CHANGED)` subscription. When events start or expire, the component won't re-render and will show stale event indicators.
- **Impact:** In practice, events change during jumps which fire `LOCATION_CHANGED` (already subscribed), so the stale window is narrow. But if viewing the panel while staying in one system across multiple day advances, event data could be wrong.
- **Suggested fix:** Add `useGameEvent(EVENT_NAMES.ACTIVE_EVENTS_CHANGED)` at the top of the component. The return value doesn't need to be used directly — the subscription alone ensures re-render when events change.
- **Confidence:** Medium
- **Found by:** Concurrency & State
