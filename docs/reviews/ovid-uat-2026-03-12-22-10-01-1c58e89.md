## Critical Issues

### [C1] Prototype property bypass in EVENT_TYPES lookup crashes save sanitization
- **File:** `src/game/game-events.js:208`
- **Bug:** `removeExpiredEvents` checks `!EconomicEventsSystem.EVENT_TYPES[event.type]` to reject unknown event types. `EVENT_TYPES` is a plain object (`static EVENT_TYPES = { ... }`), so bracket access resolves inherited prototype properties. If a tampered localStorage save contains an event with `type: "constructor"`, the lookup returns `[Function: Object]` (truthy), passing the filter. Then at line 215, `EconomicEventsSystem.EVENT_TYPES[event.type].modifiers` evaluates to `[Function: Object].modifiers` which is `undefined`. At line 218, `key in validModifiers` throws `TypeError: Cannot use 'in' operator to search for 'ore' in undefined`. Since `removeExpiredEvents` runs on every day change via `updateEvents`, this permanently bricks the save -- the game crashes on every attempt to advance time.
- **Impact:** Unrecoverable crash from a simple localStorage edit. `JSON.parse` does not filter out the string `"constructor"` -- it is a perfectly valid JSON string value.
- **Suggested fix:** Use `Object.hasOwn()` instead of bracket access for existence checks:
  ```js
  if (!Object.hasOwn(EconomicEventsSystem.EVENT_TYPES, event.type)) return false;
  ```
  Apply the same fix at line 215 and in all other `EVENT_TYPES[key]` lookups (see I1).
- **Confidence:** High
- **Found by:** Security

## Important Issues

### [I1] Same prototype lookup issue in information broker and events manager
- **File:** `src/game/game-information-broker.js:302-303` and `src/game/state/managers/events.js:75`
- **Bug:** Same root cause as C1. In the information broker, `EVENT_TYPES["constructor"]` returns `[Function: Object]`, whose `.name` property is `"Object"`. The info broker would display "Object" as the event name. In the events manager, `getEventType("constructor")` returns the Function object instead of `null` (the `|| null` fallback doesn't fire because the Function is truthy).
- **Impact:** Bad display text in info broker, potential downstream issues from non-null but wrong-type return in getEventType. Less severe than C1 since C1's crash fires first on day change.
- **Suggested fix:** Use `Object.hasOwn()` in all four locations:
  - `src/game/game-events.js:208` (the filter)
  - `src/game/game-events.js:215` (the modifier lookup)
  - `src/game/game-information-broker.js:302` (event type lookup)
  - `src/game/state/managers/events.js:75` (getEventType)
- **Confidence:** High
- **Found by:** Security

### [I2] Empty-modifier events survive sanitization and block systems from getting new events
- **File:** `src/game/game-events.js:211-223`
- **Bug:** The `removeExpiredEvents` sanitization strips modifier keys not in the current `EVENT_TYPES` definition. If an old save has an event whose stored modifier keys ALL differ from the current type definition, the sanitized event gets `modifiers: {}`. This event passes the filter (type exists, not expired) and persists in the active events list. At line 87, `activeEvents.some((event) => event.systemId === system.id)` prevents new events from spawning on that system. The empty-modifier event contributes nothing to trading (`getEventModifier` returns 1.0) but blocks the system until expiry.
- **Impact:** Systems get locked out of economic events for the remaining duration of the zombie event. Players lose trading opportunities with no visible explanation. Plausible for old saves with `supply_glut` events that had dynamically-generated single-commodity modifiers (e.g., `{ grain: 0.6 }`) that won't match any current event type's canonical modifiers.
- **Suggested fix:** After sanitization, filter out events with empty modifiers:
  ```js
  .filter((event) => Object.keys(event.modifiers).length > 0)
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling

### [I3] Duplicated travel time formula and magic number 0.5 across 3 files
- **Files:** `src/game/game-navigation.js:70`, `src/game/utils/wormhole-graph.js:104`, `src/game/mission-generator.js:87`
- **Bug:** The formula `Math.max(1, Math.ceil(distLY * 0.5))` is duplicated in three independent implementations. The `0.5` factor is not defined in `constants.js`, violating the project's explicit CLAUDE.md rule: "ALL magic numbers must go in `src/game/constants.js`." If the travel time formula ever changes, all three locations must be updated in lockstep.
- **Impact:** Maintenance hazard -- a formula change in one location but not the others would silently produce incorrect mission deadlines or navigation travel times. The two new copies in `wormhole-graph.js` and `mission-generator.js` were introduced by this branch.
- **Suggested fix:** Extract to `NAVIGATION_CONFIG.DAYS_PER_LY: 0.5` in `constants.js`. Ideally extract the entire formula to a shared pure function (e.g., in `src/game/utils/calculators.js`) and have all three locations call it.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

## Suggestions

- **[S1]** `src/game/game-trading.js:43-51` -- Comment says "All event modifiers must be > 1.0" but code checks `eventMod !== 1.0`. Not a current bug (all existing modifiers are > 1.0), but a future sub-1.0 modifier would silently trigger premium pricing. Consider changing to `eventMod > 1.0` or adding a runtime assertion. *(Logic & Correctness)*
- **[S2]** `src/game/engine/scene.js:272`, `src/game/engine/game-animation.js:170` -- Comments say "shared glow texture from stars.js" but each call creates a new GPU texture (3 identical 64x64 textures). Not a performance concern at this scale, but comments are misleading. Either cache the texture in `createStarTexture()` or fix the comments. *(Error Handling, Contract & Integration, Concurrency & State)*
- **[S3]** `src/game/mission-generator.js:55` -- BFS fallback defaults to `starData = []`, silently producing 1-day-per-hop travel times. Only affects tests with custom wormhole data. Consider removing the default to force callers to be explicit. *(Error Handling)*
- **[S4]** `src/features/navigation/SystemPanel.jsx:313-318` -- `game.getActiveEventForSystem()` called directly during render rather than through Bridge Pattern hooks. However, this is a pre-existing pattern in this component (7+ direct `game.*` calls in render), not a new violation. *(Concurrency & State)*
- **[S5]** `src/game/game-trading.js:51` -- A tampered save could set a modifier value to string `"1.0"`, bypassing strict inequality and triggering premium pricing. Requires deliberate localStorage editing in a single-player game. Fix: add `typeof value === 'number'` check in `removeExpiredEvents` sanitization. *(Security)*
