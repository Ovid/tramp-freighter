# Accessibility Audit: Tramp Freighter Blues

**Date:** 2026-03-14 14:02:30
**Commit:** 40ddb884a35fdd8cdf939d122c43634a3040007b
**Tech stack:** React 18.2, Vite 5, Three.js 0.150, plain CSS with CSS Variables
**Component libraries:** None (all custom-built)
**Files audited:** 93 (58 JSX components, 35 CSS files)
**Existing a11y tooling:** None (no eslint-plugin-jsx-a11y, no axe-core, no pa11y)
**Conformance target:** WCAG 2.2 AA (AAA noted as recommendations)

## Executive Summary

The codebase has foundational patterns in place (Modal with `role="dialog"`, NotificationContainer with `aria-live`, semantic `<button>` use in most places) but significant gaps remain. The most critical barrier is that all 6 encounter panels use non-focusable `<div>` elements for mandatory gameplay choices, completely blocking keyboard and screen reader users during encounters. Contrast failures on high-frequency HUD elements, absent focus management across view transitions, and zero `prefers-reduced-motion` support compound the issue. Estimated conformance: **partial Level A** -- several Level A criteria are not met.

## Impact Summary by User Group

- **Screen reader users:** Core gameplay is severely impacted. The starmap lacks any accessible alternative, encounter options are invisible to assistive tech, 5 dialog overlays lack `role="dialog"`, and dynamic status messages (achievements, rumors, validation errors) are not announced. Some good patterns exist (Modal has ARIA, NotificationContainer has `aria-live`).
- **Low-vision users:** White text on bright condition bars fails contrast at ~1.5:1. Danger buttons fail at ~2.4:1. Font sizes use `px` units preventing browser text scaling. Focus ring is only 1px wide. Several low-opacity text patterns are borderline.
- **Colorblind users:** Status information (safe/contested/dangerous, good/fair/poor/critical, profit/loss) uses green-yellow-red color coding. Most instances include text labels which mitigates the issue, but notification types and some visual indicators rely on color alone.
- **Motor-impaired users (keyboard/switch/sip-and-puff):** The most severe barriers exist here. All 6 encounter panels are mouse-only (div+onClick). The starmap camera controls are mouse-only. Modals lack focus traps. No focus management occurs on view transitions. useClickOutside provides no keyboard equivalent.
- **Cognitive and learning disabilities:** Notifications auto-dismiss in 2-3 seconds with no pause mechanism. Help/instructions are buried in settings. Inconsistent back/close button labeling across panels. Encounter panels present dense information without progressive disclosure.
- **Deaf and hard-of-hearing users:** No audio or video content was found, so this group is not directly impacted.
- **Vestibular and photosensitive users:** Zero `prefers-reduced-motion` support. Infinite CSS pulse animations, continuous 3D starmap rotation (defaults ON), and an unskippable 3-5 second jump animation all run regardless of OS motion preferences.

## Critical Issues (Complete Barriers)

### [C1] ~~All 6 encounter panels use non-focusable divs for mandatory gameplay choices~~ FIXED
- **File:** `src/features/danger/CombatPanel.jsx:236-390`, `PirateEncounterPanel.jsx:184-294`, `NegotiationPanel.jsx:136-306`, `InspectionPanel.jsx:189-339`, `DistressCallPanel.jsx:208-315`, `MechanicalFailurePanel.jsx:142-263`
- **Barrier:** Combat options, tactical choices, negotiation strategies, inspection responses, moral decisions, and repair options are all `<div>` elements with `onClick` handlers but no `role`, `tabIndex`, or `onKeyDown`. Keyboard-only users cannot focus on, navigate between, or activate any option.
- **WCAG:** 2.1.1 (Keyboard) -- Level A
- **Affects:** All keyboard-only, switch-access, sip-and-puff, and screen reader users. Encounters are mandatory game events that block all other interaction -- the game becomes unplayable during encounters for these users.
- **Fix:** Convert each `<div className="combat-option" onClick={...}>` (and equivalents in all 6 panels) to `<button>` elements. Alternatively, use `role="radiogroup"` on the parent with `role="radio"`, `tabIndex={0}`, `aria-checked`, and `onKeyDown` on each option. The `<button>` approach is simplest and provides keyboard access automatically.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard, Cognitive, React specialists (4/6)

## Serious Issues (Major Difficulty)

### [S1] Three.js starmap camera controls are mouse-only
- **File:** `src/features/navigation/StarMapCanvas.jsx:187,411`, `src/game/engine/scene.js:109-113`
- **Barrier:** Star selection via the 3D canvas is mouse-click only (raycasting from mouse coordinates). Camera rotation, pan, and dolly use OrbitControls configured exclusively for mouse. The starmap container div has no `tabIndex`, `role`, or keyboard handlers.
- **WCAG:** 2.1.1 (Keyboard) -- Level A
- **Affects:** Keyboard-only and switch-access users cannot navigate the 3D starmap. Star selection IS possible via the "Find Star" dropdown in CameraControls (provides a keyboard-accessible alternative path), but camera orientation has no keyboard equivalent.
- **Fix:** (1) Add `tabIndex={0}`, `role="application"`, and `aria-label="Interactive 3D star map -- use Find Star dropdown to select systems"` to the container. (2) Add keyboard handlers: arrow keys for camera rotation, +/- for zoom (supplementing existing zoom buttons). (3) Ensure the Find Star dropdown is prominently discoverable.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard specialists (2/6)

### [S2] Modal component lacks focus trap, initial focus, and focus restoration
- **File:** `src/components/Modal.jsx:62-92`
- **Barrier:** The Modal correctly has `role="dialog"`, `aria-modal="true"`, and Escape handling. However: (1) focus does not move into the modal on open, (2) Tab/Shift-Tab can escape the modal into background content, (3) focus is not restored to the triggering element on close.
- **WCAG:** 2.4.3 (Focus Order) -- Level A
- **Affects:** Keyboard and screen reader users can Tab past the modal into obscured background content. When the modal opens, they don't know it appeared. When it closes, focus is lost.
- **Fix:** On open, save `document.activeElement` and focus the first focusable element inside the dialog. Trap Tab/Shift-Tab to cycle within the modal. On close, restore focus to the saved element.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard, React specialists (3/6)

### [S3] Five dialog overlays bypass Modal component -- no dialog semantics
- **File:** `src/features/title-screen/ShipNamingDialog.jsx:39-81`, `src/features/missions/MissionCompleteNotifier.jsx:51-96`, `src/features/navigation/SystemPanel.jsx:133-146`, `src/features/upgrades/UpgradesPanel.jsx:150-210`, `src/features/narrative/NarrativeEventPanel.jsx:73-95`
- **Barrier:** These components render `<div className="modal-overlay">` (or equivalent overlay) without `role="dialog"`, `aria-modal="true"`, `aria-labelledby`, focus trapping, or Escape key handling. Screen readers do not identify these as dialogs, and keyboard focus can escape into background content.
- **WCAG:** 4.1.2 (Name, Role, Value) -- Level A; 1.3.1 (Info and Relationships) -- Level A
- **Affects:** Screen reader users are not informed these dialogs appeared. Keyboard users can Tab behind the overlays. The ShipNamingDialog input also lacks a `<label>` element.
- **Fix:** Refactor each to use the shared `Modal` component, or add `role="dialog"`, `aria-modal="true"`, `aria-labelledby` referencing the heading, focus management, and Escape handling.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard, React specialists (3/6)

### [S4] No focus management on view mode transitions
- **File:** `src/App.jsx:92,196-211,263-276`, `src/features/station/PanelContainer.jsx:25-58`
- **Barrier:** The view mode state machine (TITLE, ORBIT, STATION, PANEL, ENCOUNTER) transitions without moving focus. When entering ENCOUNTER mode, keyboard focus stays on whatever was focused before (possibly now-hidden elements). When opening a panel from StationMenu, focus stays on the menu button, not the panel.
- **WCAG:** 2.4.3 (Focus Order) -- Level A
- **Affects:** Keyboard and screen reader users lose focus context during view transitions. They must Tab through the entire page to find new content. During encounters (which are time-sensitive), they may not realize an encounter has appeared.
- **Fix:** Add `useEffect` + `useRef` patterns to focus the heading or first interactive element when each view mounts. Add `tabIndex={-1}` to panel containers for programmatic focus.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard, React specialists (3/6)

### [S5] ~~White text on bright HUD condition bars fails contrast~~ FIXED
- **File:** `css/hud.css:144,156-192`
- **Barrier:** `.condition-text` uses white (`#ffffff`) overlaid on bright colored bars: fuel `#00ff88` (~1.5:1), hull `#4a9eff` (~2.7:1), engine `#ffaa00` (~1.7:1), life support `#ff4a9e` (~3.1:1). All fail AA minimum of 4.5:1 for normal text. The `text-shadow` at line 147 helps but does not compensate.

- **Affects:** Low-vision users cannot read percentage values on condition bars -- the most frequently viewed UI element in the game.
- **Fix:** Change `.condition-text` to `color: #000000`. Black text passes on all bar colors (fuel 15.7:1, hull 7.6:1, engine 11.0:1, life support 6.7:1).
- **Confidence:** High
- **Found by:** Visual & Color specialist

### [S6] ~~Danger button white text on red background fails contrast~~ FIXED

### [M16] ~~No `prefers-reduced-motion` support anywhere in codebase~~ FIXED
- **File:** All CSS and JS files (project-wide)
- **Barrier:** Zero `prefers-reduced-motion` media queries in CSS or JS. Infinite CSS pulse animations (`.ship-reticle`, `.severity-value.emergency`), notification slide-ins, dialogue fade-ins, starmap auto-rotation, and 3D selection ring pulsing all run regardless of OS motion preferences.
- **WCAG:** 2.3.3 (Animation from Interactions) -- Level AAA; best practice under 2.2.2
- **Affects:** Users with vestibular disorders (vertigo, migraines triggered by motion) and photosensitive users.
- **Fix:** Add global CSS rule: `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }`. For Three.js, check `window.matchMedia('(prefers-reduced-motion: reduce)').matches` and disable auto-rotation default, skip jump animation phases, and use static selection ring values.
- **Confidence:** High
- **Found by:** Visual & Color, Multimedia specialists

### [M17] Notifications auto-dismiss with no user control

- **Confidence:** High
- **Found by:** Cognitive, Multimedia specialists
